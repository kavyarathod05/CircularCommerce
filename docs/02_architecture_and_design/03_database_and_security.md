# Database Design & Security Architecture

## 1. Single-Table Database Design (Amazon DynamoDB)

SecondLife Commerce utilizes a single-table design in DynamoDB to achieve sub-millisecond latencies and query related datasets in single requests.

### 1.1 Table Structure & Primary Keys

#### Users Table
- **PK**: `USER#{userId}` | **SK**: `PROFILE`
- **Attributes**:
  - `userId` (String): UUID
  - `email` (String)
  - `role` (String): customer / seller / ops
  - `trustScore` (Number): 0–100
  - `ecoPoints` (Number): Gamification
  - `location` (Map): lat, lng
  - `preferences` (Map): notifications, radius
  - `createdAt` (String): ISO timestamp
- **GSIs**:
  - `EmailIndex` (email → userId) for login lookup

#### Products Table
- **PK**: `PRODUCT#{productId}` | **SK**: `METADATA`
- **Attributes**:
  - `productId` (String)
  - `sellerId` (String)
  - `name` (String)
  - `category` (String)
  - `originalPrice` (Number)
  - `catalogImages` (List): S3 URLs
  - `createdAt` (String)

#### Returns Table
- **PK**: `RETURN#{returnId}` | **SK**: `STATUS`
- **Attributes**:
  - `returnId` (String): UUID
  - `userId` (String)
  - `productId` (String)
  - `orderId` (String)
  - `reason` (String)
  - `status` (String): initiated / inspected / routed / completed
  - `createdAt` (String)
  - `updatedAt` (String)
- **GSIs**:
  - `UserReturnsIndex` (userId, createdAt) — all returns for a user
  - `SellerReturnsIndex` (sellerId, createdAt) — all returns for a seller

#### Inspections Table
- **PK**: `INSPECTION#{inspectionId}` | **SK**: `RESULT`
- **Attributes**:
  - `inspectionId` (String): UUID
  - `returnId` (String)
  - `grade` (String): A/B/C/D
  - `damages` (List): type, severity, description
  - `fraudSignals` (List)
  - `confidenceScore` (Number): 0–1
  - `certificateUrl` (String): S3 URL
  - `imageUrls` (List): S3 URLs
  - `inspectedAt` (String)
- **GSIs**:
  - `ReturnInspectionIndex` (returnId) — inspection for a return

#### Routing Decisions Table
- **PK**: `ROUTING#{returnId}` | **SK**: `DECISION`
- **Attributes**:
  - `returnId` (String)
  - `pathway` (String): hyperlocal/marketplace/refurb/recycle/donate/liquidate
  - `reasoning` (String): Human-readable explanation
  - `sellerOverride` (Boolean)
  - `decidedAt` (String)
  - *Extensions for low-value routing*:
    - `consolidationNodeId` (String): Locker/kiosk ID
    - `nodeProximityKm` (Number): Distance to drop-off node
    - `bulkPickupWindowDate` (String): Consolidation window date

#### Marketplace Listings Table
- **PK**: `LISTING#{listingId}` | **SK**: `DETAIL`
- **Attributes**:
  - `listingId` (String): UUID
  - `returnId` (String)
  - `productId` (String)
  - `grade` (String)
  - `askingPrice` (Number)
  - `status` (String): available/sold/removed
  - `certificateUrl` (String)
  - `location` (Map): lat, lng
  - `publishedAt` (String)
- **GSIs**:
  - `CategoryGradeIndex` (category, grade) — browse by category/grade
  - `StatusIndex` (status, publishedAt) — active listings

#### Carbon Metrics Table
- **PK**: `CARBON#{returnId}` | **SK**: `METRIC`
- **Attributes**:
  - `returnId` (String)
  - `userId` (String)
  - `sellerId` (String)
  - `co2SavedKg` (Number)
  - `transportLegsAvoided` (Number)
  - `warehouseDaysAvoided` (Number)
  - `pathway` (String)
  - `calculatedAt` (String)
- **GSIs**:
  - `UserCarbonIndex` (userId, calculatedAt) — user total carbon
  - `SellerCarbonIndex` (sellerId, calculatedAt) — seller total carbon

#### Events Table
- **PK**: `EVENT#{eventId}` | **SK**: `TS#{timestamp}`
- **Attributes**:
  - `eventId` (String): UUID
  - `eventType` (String): e.g. inspection.completed
  - `entityId` (String): returnId / listingId / userId
  - `payload` (Map): Full event payload
  - `timestamp` (String)

## 2. Security Design

### 2.1 Authentication & Authorization
- **Identity Provider**: Amazon Cognito User Pools registers and authenticates all users (customers, sellers, ops).
- **Access Control**: API Gateway verifies JWT tokens generated by Cognito using Cognito Authorizers.
- **RBAC**: Enforced via API Gateway resource policies and custom IAM roles:
  - `customer`: access to initiate returns, upload images, purchase open-box listings.
  - `seller`: access to the Seller Dashboard, manage overrides, and download carbon reports.
  - `ops_refurb` / `ops_warehouse`: access to the Refurb/Warehouse management views.
- **DynamoDB Fine-Grained Access**: Enforces that sellers can only read/write returns and listings marked with their respective `sellerId`.

### 2.2 Fraud Prevention
- **AI Fraud Flags**: Multimodal analysis flags mismatched barcodes, IMEI discrepancies, and serial anomalies.
- **Trust Scores**: Users are assigned a Trust Score (0-100) based on return history and inspection outcomes. Suspicious returns flag the user account.
- **Velocity Checks**: Submitting more than 3 returns in a 7-day window flags the user for manual review and freezes automated refund triggers.
- **Eco-Registry Credits Lock**: Returns for low-value items processed through the Keep-and-Credit path are issued in green credits instead of liquid cash, closing the loop on malicious cash-out exploits.

### 2.3 Cryptographic Trust & Auditing
- **AWS KMS Signing**: Product Condition Certificates are cryptographically hashed and signed using keys managed in AWS KMS. This makes the certificates tamper-evident.
- **Audit Logs**: Every write mutation to DynamoDB triggers a stream entry captured by AWS Lambda, creating an immutable log in the Events Table for operational audits.
- **Secure Image Uploads**: Product photos are uploaded directly from clients to Amazon S3 via temporary, 15-minute expiring S3 pre-signed URLs.

### 2.4 Data Privacy & Compliance
- **PII Encryption**: Encrypted at rest using AWS KMS Customer-Managed Keys.
- **Data Deletion**: Compliance with GDPR/CCPA is handled by a scheduled Lambda cleanup job triggered upon user deletion requests.
